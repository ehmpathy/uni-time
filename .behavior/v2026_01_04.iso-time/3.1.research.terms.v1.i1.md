# research: terminology for time duality

## .question

what terms does the industry use for the duality between:
1. **absolute instants** — a specific point in history (e.g., `2024-01-15T14:30:00Z`)
2. **cyclical patterns** — a repeatable time unit (e.g., `09:00` or `January`)

---

## .findings by library/standard

### java.time (JSR-310)

| type | semantics | example |
|------|-----------|---------|
| `Instant` | absolute point in time (UTC) | machine timestamps, event logs |
| `LocalDateTime` | date+time without timezone | wall clock display |
| `LocalDate` | date without time or timezone | birthdays |
| `LocalTime` | time without date or timezone | meeting times |
| `ZonedDateTime` | instant + timezone | scheduled events |
| `OffsetDateTime` | instant + offset | API responses |

> "Instant represents an instantaneous point on the time-line... models a single instantaneous point on the time-line"
> — [Baeldung: Instant vs LocalDateTime](https://www.baeldung.com/java-instant-vs-localdatetime)

> "LocalDateTime is not a point on the time line as Instant is, LocalDateTime is just a date and time as a person would write"
> — [Java Code Geeks](https://examples.javacodegeeks.com/java-instant-vs-localdatetime/)

**key insight:** Java uses `Local*` prefix to mean "no timezone info" (wall-clock perspective)

---

### javascript temporal API

| type | semantics | example |
|------|-----------|---------|
| `Temporal.Instant` | specific point in time (UTC) | timestamps |
| `Temporal.PlainDateTime` | date+time without timezone | local events |
| `Temporal.PlainDate` | date only | anniversaries |
| `Temporal.PlainTime` | time only | daily schedules |
| `Temporal.PlainYearMonth` | year+month | billing periods |
| `Temporal.PlainMonthDay` | month+day (no year) | annual holidays |
| `Temporal.ZonedDateTime` | instant + timezone | meetings across zones |

> "The Plain data type represents a date or time that has no time zone information"
> — [MDN: Temporal](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Temporal)

> "Temporal cleanly separates plain dates, absolute timestamps, and time zones"
> — [Better Stack: Temporal Explained](https://betterstack.com/community/guides/scaling-nodejs/temporal-explained/)

**key insight:** Temporal uses `Plain*` prefix to mean "no timezone info" (wall-clock perspective)

---

### noda time (.NET)

| type | semantics | example |
|------|-----------|---------|
| `Instant` | absolute point in time | event timestamps |
| `LocalDateTime` | date+time without timezone | wall clock |
| `LocalDate` | date without time | birthdays |
| `LocalTime` | time without date | daily routines |
| `ZonedDateTime` | instant + timezone | scheduled events |
| `OffsetDateTime` | instant + offset | API data |

> "Clock-related 'point in time' values are usually best represented as Instant values"
> — [Noda Time: Type Choices](https://nodatime.org/2.2.x/userguide/type-choices)

> "A LocalDateTime is a point on a time line in a particular calendar system, but with no concept of the offset from UTC"
> — [Noda Time: Core Types](https://nodatime.org/3.0.x/userguide/core-types)

**key insight:** follows Java's convention with `Local*` prefix

---

### google CCTZ (C++)

| term | semantics |
|------|-----------|
| `Absolute time` | a specific instant in history |
| `Civil time` | YMDHMS as displayed on wall clock |

> "There are two main ways to think about time in a computer program: as absolute time, and as civil time"
> — [Google CCTZ](https://github.com/google/cctz)

> "A civil time is represented as six individual fields that represent a year, month, day, hour, minute, and second (aka YMDHMS). These six fields represent the time as defined by some local government"
> — [Google Open Source Blog](https://opensource.googleblog.com/2016/04/cctz-v20-now-with-more-civil-time.html)

**key insight:** Google uses `Absolute` vs `Civil` — cleaner than `Instant` vs `Local`

---

### system clocks (OS level)

| term | semantics |
|------|-----------|
| `Wall clock` | real-world time (can jump, NTP-synced) |
| `Monotonic clock` | steady forward progression (for durations) |

> "The wall clock is the 'real-world' clock that shows calendar dates and times"
> — [VictoriaMetrics: Go Time](https://victoriametrics.com/blog/go-time-monotonic-wall-clock/)

**key insight:** "wall clock" = human-readable time display

## .candidate terms for our duality

### for absolute instants (stamps)

| term | pros | cons |
|------|------|------|
| **`Stamp`** | evokes "timestamp", clear | "datestamp" awkward |
| **`Instant`** | industry standard (Java, Noda, Temporal) | less intuitive for non-devs |
| **`Absolute`** | Google uses it, very clear | longer, less common |
| **`Point`** | "point in time" is natural English | less specific |
| **`Mark`** | "marked on calendar" | not industry standard |

### for cyclical patterns (no timezone, repeatable)

| term | pros | cons |
|------|------|------|
| **`Onwall`** | evokes "wall clock", unique | novel jargon, requires explanation |
| **`Plain`** | Temporal API uses it | doesn't convey "cyclical" nature |
| **`Local`** | Java/Noda use it | misleading (implies locality) |
| **`Civil`** | Google uses it | less known, sounds governmental |
| **`Clock`** | "clock time" is intuitive | could confuse with system clock |
| **`Wall`** | "wall clock time" | might be too abbreviated |
| **`Cycle`** | conveys repeating nature | not industry standard |
| **`Pattern`** | cron uses "pattern" | too generic |
| **`Recur`** | conveys repetition | not a noun |

## .key observations

### 1. industry prefers prefix over suffix

- Java: `Local` + DateTime, Date, Time
- Temporal: `Plain` + DateTime, Date, Time, YearMonth, MonthDay
- Noda: `Local` + DateTime, Date, Time

our proposal inverts this:
- `IsoTimeStamp`, `IsoDateStamp`, `IsoTimeOnwall`

**question:** should we follow convention with prefix pattern instead?
- `IsoInstantTime`, `IsoPlainTime`, `IsoPlainMonth`?

### 2. "Local" is misleading

"Local" suggests geographic locality, but it really means "no timezone info."

> "LocalDateTime seems to be very similar to Instant... But there is a difference: LocalDateTime is not a point on the time line"
> — [Java Code Geeks](https://examples.javacodegeeks.com/java-instant-vs-localdatetime/)

### 3. "Plain" is clearer than "Local"

Temporal API chose `Plain` specifically to avoid the "Local" confusion.

> "The Plain data type is best utilized when you want to represent time with no information as to what time zone the time comes from"
> — [Pieces.app: Temporal API](https://pieces.app/blog/javascript-temporal-api)

### 4. no standard term for "cyclical/repeatable" units

cron expressions use individual fields, but there's no standard type name for:
- "time of day that repeats daily"
- "month that repeats yearly"
- "day of month that repeats monthly"

`PlainMonthDay` (Temporal) comes closest — it represents "March 15" without a year.

## .alternative proposals

### option 1: follow Temporal's `Plain` pattern

```ts
IsoInstant       // absolute moment (matches industry)
IsoPlainTime     // time of day (no date)
IsoPlainDate     // date (no time)
IsoPlainMonth    // month without year context
```

**pros:** aligns with JavaScript Temporal, clear
**cons:** loses "Stamp" distinction

### option 2: use `Stamp` vs `Clock`

```ts
IsoTimeStamp     // absolute moment
IsoDateStamp     // absolute date
IsoTimeClock     // time of day (wall clock)
IsoMonthClock    // month pattern
```

**pros:** `Clock` evokes wall clock naturally
**cons:** could confuse with system clock concepts

### option 3: use `Stamp` vs `Cycle`

```ts
IsoTimeStamp     // absolute moment
IsoDateStamp     // absolute date
IsoTimeCycle     // time of day (repeats daily)
IsoMonthCycle    // month (repeats yearly)
```

**pros:** `Cycle` clearly conveys repetition
**cons:** not industry standard

### option 4: use `Stamp` vs `Wall`

```ts
IsoTimeStamp     // absolute moment
IsoDateStamp     // absolute date
IsoTimeWall      // wall clock time
IsoMonthWall     // month on wall calendar
```

**pros:** "wall clock" is well-understood
**cons:** `Wall` alone feels incomplete

### option 5: keep `Stamp` vs `Onwall`

```ts
IsoTimeStamp     // absolute moment
IsoDateStamp     // absolute date
IsoTimeOnwall    // time on wall clock
IsoMonthOnwall   // month on wall calendar
```

**pros:** `Onwall` is distinctive, memorable
**cons:** novel jargon, requires learning

---

## .recommendation

based on research:

1. **`Stamp`** is reasonable for absolute instants (evokes "timestamp")

2. **`Plain`** is the most industry-aligned term for "no timezone" (Temporal API)

3. **`Onwall`** is creative but requires explanation

4. **`Clock`** or **`Wall`** might be clearer than `Onwall`

**suggested hybrid:**

```ts
// absolute instants (stamps)
IsoTimeStamp     // '2024-01-15T14:30:00Z'
IsoDateStamp     // '2024-01-15'
IsoMonthStamp    // '2024-01'
IsoYearStamp     // '2024'

// wall clock patterns (cyclical)
IsoTimeWall      // '14:30' — time of day
IsoMonthWall     // '01' — January (repeats yearly)
IsoDayWall       // '15' — 15th (repeats monthly)
```

or follow Temporal more closely:

```ts
IsoInstant       // '2024-01-15T14:30:00Z'
IsoPlainDate     // '2024-01-15'
IsoPlainTime     // '14:30'
IsoPlainMonth    // '01'
```
